# 1. 多模態：我支持，但你要明確「數學多模態」不是一般 VLM

這件事我同意，而且對數學非常重要。因為高階數學任務天然就不是純文字：

* 題目可能在 PDF 掃描檔
* 解答可能有表格、分段公式、註記
* 幾何、圖論、組合題常依賴圖形

## 不要把它做成「能看圖的通用聊天模型」

**Document-native mathematical reasoning model**

也就是三層能力：

### A. 文件解析層

* object
* relation
* constraint
* proof obligation
* unresolved symbol
* assumption scope
* diagram-grounded relation

### C. 推理控制層

模型要能決定：

* 現在是在「讀文件」還是「解題」
* 是在抽 lemma 還是做 proof search
* 是否需要回看前頁
* 是否需要引用圖形
* 是否需要將自然語言轉 formal skeleton

所以多模態不是加一個 vision encoder 就結束。
你真正要做的是：

**document parsing + math structure induction + controlled reasoning**

# 2. 資料策略大改：這是對的，而且是最該先做的

這點我非常認同。
做數學模型，如果資料策略不重構，後面所有架構改動都容易白費。

## 我建議你把資料分成五個主池

### Pool A：數學知識底座

用途：建立術語、定義、命題風格、符號語感

來源類型：

* 教科書
* 講義
* 數學百科/定義型材料
* 論文中的 preliminaries / background
* formal math library 的 human-readable 部分

這一池不是拿來訓練「做題」，而是訓練：

* definition precision
* symbol disambiguation
* statement parsing
* theorem style familiarity

---

### Pool B：程序性解題資料

用途：訓練 stepwise transformation 與解法遷移

來源類型：

* olympiad solution
* contest solution
* worked examples
* stackexchange 高品質解答
* proof-oriented教學材料

但這池要嚴格清洗，因為很多解答有這些問題：

* 跳步
* 隱含常識太多
* final answer correct 但中間不嚴謹
* 模板化口癖太重

所以不能把它當 gold proof，只能當 weak procedural trace。

---

### Pool C：形式化/半形式化資料

用途：訓練 verifier-friendly reasoning

來源類型：

* Lean / Isabelle / Coq 風格證明
* theorem / lemma / tactic trace
* semi-formal proof skeleton
* natural-to-formal aligned pairs

這池的重要性很高，因為它會決定你未來 verifier 有沒有根。

---

### Pool D：文件級資料

用途：訓練長距依賴、跨頁引用、定義回溯、定理-證明鏈

來源類型：

* PDF 講義
* 論文
* note
* textbook chapter
* scanned notes + OCR / layout parse

這池是你「文件閱讀能力」的核心。

---

### Pool E：合成資料

用途：不是灌答案，而是灌機制

只做這幾類最有價值：

* failure recovery
* strategy switch
* counterexample generation
* equivalent reformulation
* state tracking
* verifier contrast pairs
* correct vs subtly invalid proof discrimination

---

## 我不建議的資料方向

### 1. 大量低質數學 QA

很容易把模型訓練成：

* 看到熟題就背
* 看到陌生題就亂補
* 對 proof 產生虛假流暢性

### 2. 只堆 benchmark-style short problems

這會讓模型缺：

* lemma abstraction
* long-context proof dependency
* theorem reuse
* document grounding

### 3. 把 benchmark 原題直接當主要 supervised corpus

這會讓 eval 失真，尤其你後面又要拿同類 benchmark 做階段評估。

# 3. State IR 和 level contracts 全面重構：我支持，但要避免「設計得太漂亮卻學不起來」

這部分是整個系統的核心。
既然你已經決定不受 baseline 約束，那我會建議你乾脆把 IR 重定義成**數學推理作業系統**。

## 我建議新的 State IR 至少有這七層槽位

### 1. Problem Frame

* task type
* required output form
* assumptions
* domain
* target object

例如：
證明 / 求值 / 構造 / 判定 / 找反例 / formalize

---

### 2. Symbol Table

* variables
* constants
* functions
* sets
* indices
* quantifiers
* type / domain

這是避免 parse error 的基礎。

---

### 3. Constraint Graph

* equalities
* inequalities
* divisibility
* implication
* geometric incidences
* recurrence edges
* dependencies

數學很多題本質上是 constraint graph，不是字串問題。

---

### 4. Proof / Program Frontier

* current hypotheses
* current subgoals
* candidate strategies
* active branch
* abandoned branch
* unresolved obligations

這層對 search 特別重要。

---

### 5. Memory / Lemma Interface

* retrieved pattern
* applicable lemma
* confidence
* matching conditions
* mismatch notes

重點不是有沒有 memory，而是**有沒有 applicability audit**。

---

### 6. Verifier State

* local step validity
* hidden assumption risk
* gap detection
* counterexample risk
* branch consistency score

這讓 verifier 不是只有末端二元判定。

---

### 7. Control State

* continue / backtrack / reparse / switch strategy / stop
* budget remaining
* uncertainty
* escalation condition

這是整個系統從「會生成」變成「會解題」的關鍵。

---

## 新的 level contracts 可以這樣設

你原本若是線性分層，我建議改成**能力責任分層**：

### L0 Representation

讀懂題目、文件、符號、圖形，不亂解釋。

### L1 Structuring

建立 symbol table、constraint graph、document structure。

### L2 Strategy Induction

提出候選解法族，不只一條。

### L3 Search Control

選枝、回退、切換、分配算力。

### L4 Memory / Retrieval Binding

引用 theorem、例題、lemma，但要做適用性檢查。

### L5 Abstraction / Compression

把局部步驟抽成 macro / lemma / invariant。

### L6 Verification

檢查局部與整體一致性，反例試探，formal consistency。

### L7 Recovery / Repair

發現哪裡錯，重啟哪一層，而不是整題推倒重來。

### L8 Learning Signal Router

把成功/失敗 credit 精準回傳到 representation / search / memory / verifier。

這樣的 contracts 比「一般 agent pipeline」更適合數學。

# 4. 120B 分階段成長：方向合理，但這條 3-7-14-30-70-120 不一定是最佳節點

## 我比較建議的分階段意義

### Phase A：3B

目標不是分數，而是驗證：

* 新 State IR 可不可以學
* 文件解析有沒有用
* verifier 能不能 work
* search control 是否有訊號
* 資料池是否乾淨

### Phase B：8B 左右

這階段開始看：

* strategy diversity
* theorem reuse
* OCR / document robustness
* partial formalization ability

### Phase C：15B–20B

這階段可以開始看：

* 長文推理
* cross-page proof
* nontrivial olympiad performance
* memory 與 verifier 的協同

### Phase D：30B–40B

這時才有資格談：

* 較穩定高難數學
* 多路 search 有顯著收益
* Frontier-class 任務出現可觀信號

### Phase E：70B+

開始追：

* 強抽象遷移
* 新題型泛化
* 文檔/圖片/正式證明之間的統一能力

### Phase F：120B

這時應該是做：

* 最終整合
* 深 verifier
* 強 memory binding
* 多模態 fully native

* verifier 是否同步變強
* data quality 是否跟得上
* hard example mining 是否成熟
* search budget 是否被有效利用

沒有這些，120B 很容易只是更會「說得像懂」。

# 5. benchmark 納入訓練：可以，但不能是粗暴地「直接混進去」

你說 benchmarks 加入訓練，因為要階段性驗證。
這個方向我不反對，但要非常小心制度設計。

## 先講核心原則

**benchmark 可以進訓練，但 evaluation 必須重新分層。**

否則你最後不知道自己是在測：

* 能力
* 污染
* 近鄰記憶
* prompt overfit
* format familiarity

---

## 我建議做成三層 benchmark 制度

### Tier 1：Train-visible benchmark pool

這些可以進訓練，但用途是：

* format adaptation
* curriculum anchor
* early-stage sanity check
* strategy repertoire coverage

這層不能再當主要最終評估。

---

### Tier 2：Train-hidden homologous eval

跟訓練 benchmark 同分布、同風格、不同題目。
用來看：

* 近分布泛化
* 是否只是背題型模板

這層很重要，否則你看不到 overfitting。

---

### Tier 3：Strict held-out frontier eval

這層完全不進訓練，用來看真正能力：

* AIMO private-like
* unseen frontier problems
* adversarial reformulations
* cross-modal problem versions
* document-grounded theorem tasks

這才是最後要看的。

---

## benchmark 納訓的正確方式

### 方式 A：只納入「結構訊號」

例如：

* 題目類型
* solution family label
* difficulty band
* proof style
* required abstraction class

而不是整題整解直接大比例 supervised。

### 方式 B：只納入過程片段

例如：

* first representation step
* branch selection
* candidate invariant proposals
* verifier contrast pairs

### 方式 C：做 curriculum，不做終局配方

早期模型可以吃 benchmark-style examples 來建立節奏感，
後期要逐步降低權重，否則模型會過擬合「競賽題語氣」。

---

## 我不建議的 benchmark 納訓方式

* 直接把 benchmark 題與標準解大量混進主訓練
* 把 leaderboard 任務當主要 data distribution
* 不做 decontamination / homologous split / adversarial rewording

這樣到後面你看到高分，也很難判斷真假。
