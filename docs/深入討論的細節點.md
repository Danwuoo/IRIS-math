## 一、現在就要深入討論並盡快封板的

### 1. 資料憲法的「可執行版本」

不是再談方向，而是要把 `docs/07` 補到能直接驅動資料工程。現在文件已經定了五大 pool、Tier 1/2/3 benchmark 制度、canonical parse artifact、provenance 和 contamination 要求；但還沒看到你們把它落成「每個 stage 的實際配比、清洗規則、拒收規則、去重指紋、parser provenance、formalizer provenance」的執行表。這一塊不先定，後面所有訓練都會漂。

你們要討論的細節包括：

* Pool A/B/C/D/E 在 P1、P2、P3 的實際 token 或 sample 配比
* Pool B 的 weak procedural trace 清洗準則
* Pool D 的 PDF/DOCX/image/scanned note 收錄標準
* contamination 指紋粒度是 document-level、theorem-level、problem-level 還是多層並行
* parser / OCR / formula parser 的版本與 provenance 欄位怎麼落盤

### 2. Benchmark 納訓制度的家法

方向已經定了：benchmark 可以進訓練，但不能粗暴混入，且必須三層分流。現在該深挖的是「每個 benchmark family 到底怎麼進 Tier 1」。AIMO、Omni-MATH、miniF2F、FrontierMath 的 train-visible granularity、homologous split、strict held-out source、decontam 方法，都要具體化，不然制度只是口號。

要討論的細節包括：

* 哪些 family 只允許 process fragments，哪些能進 structural labels
* Tier 1 是否允許 theorem statements、proof-shape fragments、difficulty bands
* homologous eval 如何建：同題型、同定理族、同 proof pattern，還是哪一種
* FrontierMath 是否完全 Tier 3，或要不要另建 derivative family
* benchmark 納訓之後，調參時哪些 eval surface 不能看

### 3. State IR 的「細粒度 schema」

`docs/02` 已經把七個 slot group 定下來了：PF / SY / CG / FR / LM / VS / CS。接下來要談的不是要不要這七個，而是每個 slot 裡到底長什麼樣、更新單位是什麼、哪些欄位是必填、哪些可以延後。

最值得深入的點：

* PF 裡 task type、required output、assumptions、source anchors 的精確欄位
* SY 的 scope / binding / type / unresolved reference 如何表示
* CG 是 typed edges 還是 relation tuples
* FR 裡 branch、subgoal、obligation 的最小單位
* LM 的 applicability audit 用什麼欄位表示
* VS 的 local validity / gap / counterexample / consistency 是否拆成多物件
* CS 的 budget、stop、switch_strategy、reparse 是 symbolic action 還是 learned logits 對應

### 4. Level contracts 與內部模組劃分

架構文件和 contracts 已經把外部接口固定在 L0-L6，而且第一輪 v2 明確要求不要改這個外部 numbering。真正要深談的是：你之前提的 L7 Recovery / L8 Learning Signal Router，現在是要以**內部子模組**形式掛在 L3 / L6 之下，還是要在 code 裡另立 internal modules 但不暴露 external ids。

這裡要定的細節：

* L3 是否內含 repair scheduler、budget allocator、branch controller 三個子頭
* L6 是否內含 verifier aggregator、credit router、calibration head
* 什麼算 mounted implementation，什麼算 stub
* diagnostics 要輸出到什麼粒度，才能支撐 failure attribution

## 二、P1 之前必須定下，但可邊做邊修的

### 5. Multimodal document pipeline 的 canonical path

多模態方向已經確立，而且不是 generic VLM，而是 document-native mathematical reasoning。現在真正該深談的是 ingestion 到 projection 的落地接口。資料憲法已經要求先變成 `math_document_record/v1`，再投影到 State IR；這裡的工程細節會直接決定你之後 document-grounded reasoning 的上限。

要討論的點：

* PDF / DOCX / image / scanned_note / diagram 的 parser family 各自用什麼 backend
* theorem / proof / lemma / definition 的 semantic unit 切分怎麼做
* formula block 與 clean text block 的對齊
* page anchor / bbox / reading_order 的 canonical serialization
* diagram relation 何時只是 candidate，何時能進 CG
* parse failure 怎麼映射到 `F_REP`

### 6. Verifier stack 的實際 backend mix

文件已經定了 verifier 的四層：local validity、proof-gap tracking、counterexample probes、formal checker bridge，也明確區分 training-time labels 與 eval-time evidence。接下來要深入談的，是你們到底想先做哪幾層、哪些是 P1 必須、哪些是 P2 才值得上。

要討論的點：

* local validity 是 classifier、sequence tagger、還是 step-level checker
* gap detector 是用 learned head 還是規則輔助
* counterexample probe 是 symbolic-first 還是 neural proposal + checker
* formal bridge 先接 Lean，還是先做半形式化 skeleton
* false accept / false reject 的優先級與 block 條件
* verifier evidence 的 logging schema 與 provenance

### 7. Failure credit 與 recovery policy 的訓練方式

`failure.credit`、L6→L3 的方向性、targeted recovery，這些在文件中已經成為核心契約。下一步要談的是：這些東西是怎麼 supervision、怎麼學、怎麼驗證「沒有 collapse」的。

具體細節：

* `failure.credit` 標註從哪裡來：人工、verifier-derived、self-consistency、synthetic contrast
* recovery policy 的 loss 怎麼設
* search.recovery_precision 怎麼量
* 多層 blame 下，先修哪一層的 policy 是否 learned
* 什麼情況允許 rerun，什麼情況必須 targeted repair

### 8. Eval philosophy 的 blocking 指標

你已經不打算讓 benchmark 分數單獨定義成功，這點文件也完全一致。現在要深談的是：P1、P2、P3 各自的 blocking metrics 是什麼，哪些是看板指標，哪些是 hard gate。

至少要定：

* P1 的硬門檻：parser usefulness、verifier stability、contamination audit、benchmark disclosure、failure.credit alive
* P2 的硬門檻：strategy diversity、theorem reuse、OCR robustness、partial formalization gains
* 什麼叫 verifier collapse，什麼叫 benchmark lock-in
* document grounding score 與 proof validity score 的最低要求
* paired reformulation / adversarial reformulation 的 eval set 如何建

### 9. P1→P4 的 promotion story

你的資源路線有 H100 / H200 NVL / H200 SXM / B200，文件也已經把 P1~P4 profile family 定好了。但真正要討論的是：你是不是堅持 3-7-14-30-70-120 這條節點鏈，還是改成能力成熟度驅動。從目前文件看，promotion 的核心不是參數，而是 readiness。

這裡要深談：

* 哪些東西必須 frozen 才能 scale：tokenizer、IR schema、tier policy、parser provenance schema
* 哪些東西允許在 P2/P3 才重構：verifier backend、retrieval memory、multimodal adapter
* 7B/14B 是否都要做，還是直接 3→14→30 更合理
* 70B 前要不要先證明 memory-verifier coupling 有用
* 120B 前必須先滿足哪些 readiness blockers

## 三、可以延後，但遲早要啟動的

### 10. Code 對 docs 的遷移順序

目前狀態很明確：文件主幹已完成，但下一步就是把 `src/` 的 State IR / metrics / regression 實作對齊 active docs；而且你們自己也標了 **不確定**：目前 src 很可能還殘留 baseline 的 T/G/O/R/X/M scaffolding。這件事本身就是一個需要單獨設計的討論題。

這裡要討論的點：

* 遷移順序是否為 `schema/state_ir.py` → `levels/` → `metrics/` → `regression/`
* baseline adapter 保留多久
* tests 哪些先改成 v2 contract，哪些先容忍 transitional behavior
* ARC probes 保留為 compatibility signal 的範圍與期限

### 11. Archive probes 的角色邊界

`docs/11`、`docs/12` 已被明確降成 archive，不是 active authority；但 legacy ARC probes 仍可能暫時作為 compatibility signal。這件事不能一直模糊，不然很容易偷偷把舊世界觀帶回來。

要談的點：

* ARC probes 在 P1/P2 還要不要保留
* 它是 report-only 還是某些 suite 的 observe signal
* 何時正式退休
* 如何避免 ARC probe 改善掩蓋 math-native 目標停滯

### 12. Training run governance 與 reproducibility 細節

當你開始上長訓練、multimodal parser、verifier-derived labels，resume 一致性和 provenance 會變得非常重要。現在文件已經明定 journal、checkpoint、RNG、runtime lock、parser/formalizer/verifier provenance 的字段；接下來要談的是實作成本與最小可行落地。

重點包括：

* P1 階段先做到哪些 journal 字段
* parser/verifier provenance 是否每 segment 都記
* document-derived slice identity 如何穩定
* S8 在哪個 phase 真正打開 blocking
